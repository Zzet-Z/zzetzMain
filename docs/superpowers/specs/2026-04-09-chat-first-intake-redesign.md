# Chat-First 需求梳理与 Token 管理重构设计

## 1. 文档目的

本文档定义现有“阶段式需求梳理页”向“ChatGPT 风格单聊天窗口”重构的目标形态，并补充受控 token 发放、后台管理页、文档修订链与超时失效规则。

本次设计是一次产品行为重构，不是局部样式调整。实现后，用户感知应从“带阶段条的表单化梳理流程”切换为“受邀请后进入一个自然连续的中文需求对话”。

## 2. 改动目标

### 2.1 目标

- 前台需求梳理页收敛为单聊天窗口，交互参考主流 LLM 聊天产品
- 用户消息在右侧，LLM 回复在左侧
- 附件上传入口内嵌在输入区附近，不再独立展示复杂侧栏
- 不再向用户展示模板、风格、定位、内容、功能、生成等阶段模块
- 所有需求沟通都由真实 LLM 在对话中完成
- LLM 在时机成熟时主动提出是否生成最终需求文档，用户确认后再归档
- 会话完成后结束当前 token，并自动签发新的修订 token
- 修订 token 可基于上一版最终文档继续修改，但不回放旧聊天记录
- 后台提供受控 token 签发与 token 数据查看能力
- 首页保留现有产品介绍职责，但不再匿名创建会话

### 2.2 非目标

- 不引入正式账号体系
- 不引入支付、订阅、多租户权限
- 不增加后台人工接管对话
- 不做 WebSocket，继续使用轮询
- 不在本轮实现后台直接编辑最终文档
- 不物理删除 24 小时未完成 token 的数据库记录

## 3. 核心设计原则

### 3.1 用户体验原则

- 用户看到的是纯聊天，不是问卷，不是步骤条
- 系统应表现为“理解与追问”，而不是“按字段采集信息”
- 用户要能感知 LLM 正在思考，避免长时间无反馈
- 最终产物必须是中文摘要与中文需求文档，不是原始聊天记录

### 3.2 工程原则

- 继续复用现有 Flask + SQLite + 轮询架构
- 保留并发队列能力，但状态机改为轻状态机会话引擎
- token 由管理员发放，不允许匿名用户自行创建
- completed token 不可继续发送消息
- 修订链必须能追溯首版文档与后续版本关系

## 4. 新产品行为总览

### 4.1 前台访客侧

用户只能通过管理员手动发放的 token 进入应用。

进入后页面直接展示一个深色、极简、ChatGPT 风格的聊天页。页面主体是：

- 顶部极简状态栏
- 中部消息流
- 底部输入区与附件上传入口

首轮 token 的第一页应显示一条欢迎消息，引导用户直接描述自己的网站目标、目标用户、想展示的内容、喜欢的风格和参考案例。

修订 token 的第一页应显示：

- 一条新的欢迎消息
- 上一版需求摘要

但不回放上一轮聊天记录。

欢迎消息不由 LLM 即时生成，而由后端直接返回固定文案模板，以保证稳定、一致并节省 token。建议提供两个独立模板文件：

- `backend/app/prompts/welcome_initial.md`
- `backend/app/prompts/welcome_revision.md`

### 4.2 LLM 对话侧

LLM 不再依赖用户可见的阶段推进。它通过新的系统提示词在对话中完成以下任务：

- 持续梳理用户的网站需求
- 主动识别当前信息缺口
- 用中文追问并帮助用户表达
- 适时总结已经明确的信息
- 通过结构化协议告知后端“继续对话”或“已经可以提议生成文档”
- 用户确认后，输出最终文档生成阶段所需内容

### 4.3 后台管理侧

后台是一个单独的管理页，通过管理员 token 保护。

后台提供两类核心能力：

- 签发访客 token
- 查看各 token 的状态、产出与修订关系

## 5. 会话状态模型

本次重构仍保留状态机，但只保留会话级轻状态，不再保留面向用户展示的六阶段状态。

### 5.1 Session 状态

- `queued`
  已进入等待队列，尚未占用 LLM 处理槽位
- `active`
  正在处理一次 LLM 请求
- `awaiting_user`
  最近一次 assistant 回复已完成，等待用户继续发送消息
- `generating_document`
  用户已确认生成最终需求文档，系统正在归档生成
- `completed`
  本轮需求文档已完成，当前 token 不允许继续对话
- `failed`
  当前轮处理失败，且重试已耗尽
- `expired`
  token 已失效，不再可访问

### 5.2 状态迁移原则

- 发送消息前，如会话可用，则进入 `queued` 或 `active`
- LLM 回复成功后，进入 `awaiting_user`
- 用户确认生成最终文档后，进入 `generating_document`
- 文档落盘成功后，进入 `completed`
- 三次 LLM 调用仍失败后，进入 `failed`
- 满足 24 小时失效策略的未完成 token 进入 `expired`

`generating_document` 仍应占用 LLM 并发槽位，因为该阶段本质上仍在消耗真实模型资源。5 个活跃并发限制同时约束普通对话生成和最终文档生成。

## 6. Token 生命周期与修订链

### 6.1 Token 模型

本设计采用“同一个 token 既是访问凭证，也是具体会话 token”的单层模型。

管理员签发 token 后，用户拿这个 token 直接进入前台聊天页。

### 6.2 完成后的 successor token

当前 token 进入 `completed` 后：

- 本轮会话结束
- 用户不能再用该 token 继续追问
- 系统自动签发一个新的 token
- 这个新 token 用于未来基于当前最终文档继续修改

successor token 必须在“最终文档落盘成功”的同一事务内同步创建，而不是等用户下次访问时懒创建。这样可以避免重复创建，并确保前端在轮询到 `completed` 时立刻拿到修订入口。

### 6.3 修订链行为

新 token 打开后：

- 前台只显示新的开场白和上一版需求摘要
- 不回放旧聊天记录
- LLM 背后自动读取上一版最终文档作为上下文
- 新生成的需求文档需要与上一版文档建立父子关系

系统必须能追踪：

- 首版文档
- 第 N 次修订
- 每次修订对应的 token 与来源文档

## 7. 超时与失效规则

### 7.1 5 分钟无响应

如果一个未完成 token 在最近 5 分钟内没有新交互，并且仍处于队列竞争或处理中语义内，则该 token 应退出当前对话资源链路并释放并发占位。

此规则的目标是释放有限的 5 个活跃会话资源，而不是直接让 token 永久失效。

建议行为：

- `queued` 会话超时后退出等待队列，恢复为 `awaiting_user`
- `active` 会话在当前请求结束后不再保留处理资源，占位释放，状态回到 `awaiting_user`

5 分钟规则不直接写入 `expired`。

### 7.2 24 小时未完成

如果一个 token 在创建后 24 小时内未完成最终需求文档，则该 token 标记为 `expired`。

此处采用逻辑失效，不物理删除数据库中的数据记录。后台仍应可查看该 token 的历史数据。

### 7.3 访问 expired token

前台访问已失效 token 时，应返回清晰中文提示，例如：

> 这个整理链接已经失效，请联系管理员获取新的入口。

## 8. LLM 交互设计

### 8.1 前端 typing 态

当用户发送消息后，前端立即在左侧插入一个 assistant typing 占位气泡。

展示形式可以是：

- `...`
- 或附带“正在思考”

只要用户能明显理解“模型正在处理”，即可满足要求。

### 8.2 重试机制

一次用户消息触发的 LLM 调用，最多允许三次尝试。

行为要求如下：

1. 第一次调用失败
   - 不立即将整个会话标记为失败
   - 向用户返回一条弱提示，表达“我还在深度思考，请稍后”
   - 系统立即自动发起下一次重试
2. 第二次调用失败
   - 继续自动重试
   - 可以不再重复插入多条失败提示，避免消息流污染
3. 第三次调用失败
   - 本次消息请求判定彻底失败
   - 会话状态落为 `failed`
   - 返回清晰中文错误提示

第一次失败给用户的“深度思考中，请稍后”更适合作为轻量系统提示或临时状态，而不是一条和普通 assistant 正文等价的永久消息，以减少消息流污染。

### 8.3 对话协议

“是否可以生成最终需求文档”不能依赖前端匹配自然语言关键词，必须由后端和模型之间使用机械协议。

建议 LLM 每次回复都返回一个结构化 envelope，至少包含：

- `assistant_message`
- `conversation_intent`

其中 `conversation_intent` 至少支持：

- `continue`
- `ready_to_generate`
- `final_document`

推荐实现方式：

- 后端要求模型输出 JSON
- 后端负责解析并校验结构
- 前端只消费后端返回的标准字段，不自己猜测模型意图

如果模型返回内容无法解析为合法 JSON：

- 不计入三次 LLM 失败重试
- 后端将原始文本整体视为 `assistant_message`
- `conversation_intent` 默认回落为 `continue`
- 后端记录协议解析异常日志，便于后续调整 prompt

当 `conversation_intent=ready_to_generate` 时：

- 后端将该状态显式返回给前端
- 前端展示“开始生成最终需求文档”的确认按钮
- 只有用户点击确认后，后端才进入 `generating_document`

### 8.4 对话引导原则

LLM 在整个会话中应遵循以下原则：

- 只用简体中文交流
- 不能假装知道用户没说过的信息
- 不把内部阶段概念暴露给用户
- 当用户表达模糊时，优先帮助用户拆清“目标、受众、内容、风格、功能、边界”
- 当信息已经足够时，先征求确认，再生成最终文档

## 9. System Prompt 设计

### 9.1 Prompt 来源原则

新的系统提示词应以 superpowers 中 brainstorming 的核心思想翻译为中文为基础，再额外补充“帮助普通用户搭建个人网站需求”的专用指令。

这里的“引用 brainstorming”不是指照搬其 agent 过程控制，而是抽取其产品对话原则，例如：

- 先理解用户目标，再给建议
- 一次只聚焦一个关键问题，避免让用户负担过重
- 用自然对话逐步澄清，而不是一次性抛出大量问题
- 在形成结论前先确认理解是否正确

### 9.2 Prompt 内容要求

必须新增明确的 prompt 文件路径，而不是沿用旧的阶段型 prompt 组合。建议至少包含：

- `backend/app/prompts/chat_system.md`
- `backend/app/prompts/welcome_initial.md`
- `backend/app/prompts/welcome_revision.md`
- `backend/app/prompts/render_final_document.md`

其中 `chat_system.md` 应覆盖以下要求：

- 你是一个帮助普通中文用户梳理个人网站需求的 AI 助手
- 你要像一位懂沟通、懂网站策划、懂审美的产品顾问一样对话
- 你需要主动帮助用户提炼：网站目标、用户对象、信息结构、视觉风格、核心功能、信任建立方式、行动转化方式
- 当用户说得笼统时，你要继续追问，但一次只追一个最重要的问题
- 当信息足够形成文档时，你要先询问用户是否开始整理最终需求文档
- 只有在用户明确同意后，你才可以进入最终文档输出
- 输出内容必须始终为简体中文
- 你的输出必须包含可机读的结构化字段，以便后端判断当前是继续追问还是进入文档确认阶段

### 9.3 Prompt 初稿结构

`chat_system.md` 至少应按以下段落结构组织：

1. 角色定义
2. 回复语言与用户对象约束
3. 对话方式约束
4. 个人网站需求梳理重点
5. 何时提议生成最终文档
6. 结构化输出协议

其中“结构化输出协议”应明确要求模型输出 JSON，并包含 `assistant_message` 与 `conversation_intent`。

### 9.4 最终文档生成要求

最终文档仍由真实 LLM 生成，并在服务端落盘。

文档生成时应使用：

- 本轮会话消息
- 本轮附件信息
- 若当前 token 为修订轮，则附加上一版最终文档

最终文档应至少包含：

- 项目目标
- 目标用户
- 页面/模块建议
- 内容建议
- 视觉方向
- 功能需求
- 参考附件
- 其他备注与约束

## 10. 数据模型调整

### 10.1 SessionRecord

保留 `SessionRecord` 主表，但语义调整为“受邀请 token 对应的一轮会话”。

建议新增或重定义以下字段：

- `token`
- `status`
- `origin_session_token`
- `previous_document_id`
- `next_session_token`
- `admin_note`
- `last_activity_at`
- `last_user_message_at`
- `active_started_at`
- `completed_at`
- `expires_at`
- `last_error`

### 10.2 MessageRecord

保留消息表，但去掉阶段依赖。

建议保留：

- `session_token`
- `role`
- `content`
- `created_at`

建议新增：

- `delivery_status`

其中 `delivery_status` 可以支持：

- `final`
- `retry_notice`
- `system`

### 10.3 AttachmentRecord

保留附件表，继续与 token 绑定，用于文档生成和对话参考。

### 10.4 DocumentRecord

文档表需要能表达修订链关系。

建议新增字段：

- `parent_document_id`
- `root_document_id`
- `revision_number`

## 11. 前台接口设计

### 11.1 `GET /api/sessions/<token>`

返回：

- token 是否有效
- 当前会话状态
- 最近一屏消息列表，而不是全量消息
- 附件列表
- 当前文档状态
- 若存在上一版文档摘要，则一并返回
- 若当前会话已完成，则返回 successor token

建议默认仅返回最近 `50` 条消息，并提供：

- `has_more`
- `oldest_message_id`

轮询场景不做全量消息回传。

### 11.2 `GET /api/sessions/<token>/messages`

用于向上分页加载历史消息。

建议参数：

- `before_id`
- `limit`

建议默认 `limit=50`。

### 11.3 `POST /api/sessions/<token>/messages`

输入：

- 用户消息内容

返回：

- assistant 消息或中间重试提示
- 当前状态
- 是否需要轮询
- 若处于排队中，返回队列位置
- 若模型输出 `ready_to_generate`，返回前端确认文档生成所需的标准状态字段
- 若请求已触发最终文档生成，也应返回对应状态

### 11.4 `POST /api/sessions/<token>/attachments`

保持现有上传思路，但前端入口移到输入区。

### 11.5 `GET /api/sessions/<token>/document`

返回当前轮最终文档内容。

## 12. 后台接口与后台页设计

### 12.1 鉴权方式

后台不做正式登录页。

所有 `/api/admin/*` 路由使用环境变量中的管理员 token 保护。访问时使用标准 Bearer 头：

- `Authorization: Bearer <admin_token>`

环境变量名明确为：

- `ADMIN_TOKEN`

### 12.2 后台核心接口

- `POST /api/admin/tokens`
  - 手动签发一个新的访客 token
  - 支持填写备注
  - 支持基于某份已完成文档创建修订 token

- `GET /api/admin/tokens`
  - 返回 token 列表
  - 包含状态、最近活跃、消息数、附件数、文档状态、修订关系

- `GET /api/admin/tokens/<token>`
  - 返回单个 token 的详细信息
  - 包含文档摘要、文档状态、修订链、失败原因、successor token

- `POST /api/admin/tokens/<token>/revoke`
  - 由管理员手动撤销一个 token
  - 将 token 状态置为 `expired`
  - 已完成 token 的历史文档仍保留可查看

### 12.3 后台管理页信息结构

后台先做成一个单页 dashboard，避免过度扩张。

页面分三块：

- 管理员 token 输入区
- token 签发区
- token 列表与详情区

token 列表每行至少展示：

- token
- 备注
- 状态
- 是否首轮或修订轮
- 最近活跃时间
- 消息数
- 附件数
- 文档状态
- successor token 是否已生成

详情区至少展示：

- 当前 token 基本信息
- 上一版文档关系
- 根文档关系
- 最终文档摘要
- 最终文档内容入口
- 失败原因或失效原因

## 13. 前端页面结构

### 13.1 聊天页

聊天页应遵守 Apple 深色基调，但不做营销型长页面结构。

视觉要求：

- 深色背景为主
- 留白克制但明确
- 强调色仅用于发送按钮、焦点态、关键交互
- 卡片与输入区保持轻微层次，不做厚重阴影
- 中文文本优先可读性，不机械套用英文负字距

页面结构：

- 顶栏
- 消息滚动区
- 底部固定输入区

移动端优先要求必须显式保留：

- 输入区在手机视口下固定底部
- 消息滚动区在移动端占据主要可视高度
- 附件按钮、发送按钮和输入框在窄屏下不得挤压到不可点击
- 用户在移动端上传附件后，文件名反馈不能因布局换行而破坏输入区结构

### 13.2 首页

首页保留现有五段式产品介绍职责，不在本轮重构中推翻其信息架构。

但首页 CTA 行为需要调整：

- 不再直接创建匿名 session
- 改为引导用户输入 token，或使用管理员发放的访问链接进入聊天页
- 如果产品最终采用“带 token 的直达链接”，首页也至少要提供一个显式的 token 入口，而不是默认任何人都可创建会话

### 13.3 后台页

后台页应比访客页更工具化，但仍维持统一视觉体系。

不追求复杂控制台风格，以清晰表格和详情面板为主。

## 14. 验证要求

### 14.1 后端

- token 有效性校验测试
- completed token 禁止继续消息发送测试
- successor token 自动签发测试
- 修订 token 自动引用上一版文档上下文测试
- 5 分钟资源释放与 24 小时失效测试
- LLM 三次重试测试
- 后台管理员 token 鉴权测试

### 14.2 前端

- 聊天页基础渲染测试
- 用户消息在右侧、assistant 消息在左侧的界面测试
- typing 态测试
- 输入区附件入口测试
- completed / expired / failed 状态下输入区禁用测试
- 后台管理页 token 列表与签发流程测试
- 构建通过

### 14.3 浏览器验收

前端实现完成后，必须继续使用真实浏览器做 E2E 验证，至少覆盖：

- 访客通过有效 token 打开聊天页
- 发送消息并看到 typing 态
- 上传附件
- LLM 主动提出是否生成最终文档
- 用户确认后会话完成
- 页面展示新的修订 token
- 修订 token 打开后显示新的开场白和上一版摘要
- 后台可看到已签发 token 和对应文档状态

## 15. 迁移策略建议

本次推荐采用“保留现有主表、重写行为语义”的最小重构方案。

不建议新起一套完全独立的 thread 系统，否则会在当前稳定链路之外引入额外复杂度。

建议做法：

- 前端直接替换 session 页结构
- 后端弱化或移除六阶段 prompt 路径依赖
- 将摘要提取职责从“每阶段结构化提取”改为“最终文档生成与必要摘要输出”
- 在不破坏 SQLite 数据访问层的前提下扩展会话和文档字段

## 16. 风险与待注意事项

- 当前已有测试大量依赖阶段语义，重构时需要同步改写测试，而不是兼容一套旧 UI 壳
- 如果过多复用旧阶段字段但又不真正使用，会导致行为含混
- completed 后自动签发 successor token 的时机必须明确，避免重复创建
- 5 分钟资源释放与 24 小时失效需要统一定义“最近活跃时间”，否则前后端容易理解不一致
- 后台管理员 token 如果直接放前端本地存储，需要接受它只是 MVP 级别保护

## 17. 结论

本设计将当前产品从“阶段式需求梳理工具”重构为“受邀请访问的 ChatGPT 风格中文网站需求助手”。

它保留真实 LLM、轮询、SQLite、并发控制和最终文档生成这些既有技术边界，但把前台交互体验收敛为纯聊天，把 token 从“匿名 session 标识”升级为“管理员签发的受控访问凭证”，再通过 successor token 建立需求文档修订链。

这满足以下核心诉求：

- 用户只看到自然连续的聊天
- token 使用权由管理员控制
- LLM 能持续沟通并在合适时机提出生成文档
- 完成后自动进入下一轮可修订链路
- 后台可对 token 与文档产出进行最小但完整的管理
